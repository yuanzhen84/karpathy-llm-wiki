收藏夹吃灰多年后，我用 Obsidian + Claude Code + html-anything 搭了一个 Karpathy 式个人知识库

受 Karpathy LLM Wiki 启发，我把自己的 Obsidian 知识库改造成了一套"Markdown 源文件 + AI 维护 + HTML 成品输出"的个人知识生产流水线。

开篇：一个"收藏党"的自白

我得先坦白一件事：我是一个重度收藏党。

看到一篇好文章，第一反应不是读，是收藏。"这篇太有价值了，先存着，周末慢慢看。"

然后呢？再也没有然后了。

我的浏览器书签栏里躺着上百篇"回头再看"的技术文章。我的 Obsidian Inbox 里堆满了剪藏的网页——每一条都带着当时的热血沸腾，"这个观点太妙了""这个思路可以结合到我的项目里"。

但当我真的想用这些知识的时候，问题来了：

当初那个让我心动的观点是什么来着？是在哪篇文章里看到的？我当时脑子里闪过的那个灵感，到底是什么？

三条追问，能劝退我三次。最终的结果是：关掉文件夹，算了。

我觉得这事不赖我懒。知识从"看过"到"能用"，中间的鸿沟远比我们想象的大。你得整理、提炼、关联、内化……这个过程太苦了，绝大部分人（包括我）都倒在了第一步。

直到我遇到了 Karpathy 的 LLM Wiki 概念，又撞上了 Obsidian + Claude Code + html-anything 这个组合。

事情开始变得不一样了。

Karpathy 到底说了什么？一个让 AI 从"临时工"变成"全职管理员"的方案

Andrej Karpathy 在他的 GitHub Gist 里提出了一个看似反常识的思路：

与其每次查询时让 LLM 临时读材料、临时综合答案，不如让 LLM 提前把材料"编译"成一组人类可读、可编辑、可链接的 Markdown Wiki。

翻译成人话就是：

传统 RAG 方案里，AI 是个临时工。你问它一个问题，它才吭哧吭哧去翻原始材料，翻完了给你个答案，然后这事就翻篇了。下次你再问，它又得重头翻一遍。

而 LLM Wiki 的做法是：给 AI 配个"全职图书管理员"。材料一进来，AI 就先整理好——提炼核心观点、建立交叉引用、标记来源出处、标注待验证问题。此后你有任何问题，它先查整理好的 Wiki，只有 Wiki 不够用时才回到原始材料。

这个转向看起来简单，但它的影响是结构性的：

传统 RAG LLM Wiki
AI 角色 临时检索工 全职知识管理员
知识沉淀 一次性的对话答案 持续积累的 Wiki 页面
可编辑性 黑箱，人没法改 Markdown，人可以直接改
重复劳动 每次查询重新理解材料 只编译一次，后续复用
成本 token 浪费在重复上下文 token 花在结构化编译上

更重要的是：好答案不会死在聊天记录里了。

有过这样的体验吧？花了一下午和 AI 对话，摸索出一个绝佳的方案。关掉对话窗口，三个月后想复用——找不到了。就算找到了，模型也记不得上下文，你得重新教一遍。

LLM Wiki 把"有价值的输出"从对话记录升级成了可持续维护的知识资产。每次对话产生的新判断，都可以回写到 Wiki 页面。下一次，AI 已经"记住"了。

我的"收藏-吃灰"闭环，终于被打破了

讲实话，Karpathy 的原始方案给我开了个脑洞，但真正让我兴奋的，是我自己的组合拳——Obsidian + Claude Code/Codex + html-anything。

这三样东西正好卡在"收藏→吃灰"这个闭环的三个断点上。

第一步：收藏的时候顺手打个标签

以前收藏一篇文章，我需要"找时间认真整理"。这个心理门槛太高了——"认真整理"意味着要看完整篇、提炼核心、写笔记……对一篇随手刷到的文章来说，这个成本远远超过了它带给我的价值。

所以结果是：要么不收藏（可惜），要么收藏了不动（更可惜）。

有了 Obsidian 的浏览器剪藏插件，我的流程变成了：

  1. 看到好文章 → 一键剪藏到 Inbox
  2. 顺手写一两句当下的感受和灵感
  3. 打上标签，粗略关联到某个主题

五分钟搞定。不追求完美整理，只追求"把当时的灵光一闪记录下来"。

这一步的价值太大了——知识是会被遗忘的，但感受和灵感是有时效性的。收藏时那瞬间的"好妙啊"如果不记录，三天后再看原文，你只会觉得"嗯还行"，完全找不回当初的兴奋感和切入角度。

第二步：Claude Code 帮我"重读"和"提炼"

当我在某个领域积累了 5-10 篇相关文章后，真正的乐趣开始了。

我打开 Claude Code，告诉它：

"我有这几篇文章的剪藏和当时的批注，我想梳理一下关于 X 领域的主要流派和方法论。我不需要你复述原文，我想表达的观点是 Y。帮我找论据支撑，也可以结合我的实践经验生成案例。"

然后 Claude Code 会：

  • 读取我的 Inbox 剪藏和批注
  • 理解我的观点倾向
  • 检索相关领域的既有 Wiki 页面
  • 给我一份有出处、有逻辑、有个人观点的草稿

这个过程的本质是什么？是 LLM 帮我完成了"重读+提炼+关联"这三个我以前最不想干的脏活。

我不需要自己去翻那 10 篇文章找论点——LLM 已经读过了。我不需要自己组织逻辑结构——LLM 根据我的思路帮我搭了框架。我只需要做我最擅长的事：判断、调整、注入个人见解。

第三步：html-anything 让我不再被模型"绑架"

这一步是最让我爽的。

之前我用 NotebookLM 也做过类似的事——把材料丢进去，让它生成信息图、播客、PPT。出来的东西确实好看。

但有个致命的问题：模型生成的"图片、视频、音频"是一次性的。不可变。不可改。

我的思路是会迭代的。今天我觉得"这个角度很对"，下周读到新材料后可能就变成"不对，应该是那样"。但 NotebookLM 生成的播客改不了，PPT 改不了——我只能重新生成一个全新的版本。模型没有记忆，我每次都得重新创建任务、重新上传材料、重新迭代好几轮，才有可能得到一个差不多的东西。

核心矛盾是：我的思想是活的，但生成物是死的。

html-anything 彻底扭转了这个局面。

我是一个 web 开发出身的程序员，HTML 对我来说就跟呼吸一样自然。html-anything 提供了丰富的模板——docs-page、doc-kami-parchment、deck-swiss-international……我只需要把 Markdown 丢进去，选个模板，它就能帮我生成漂亮的网页。

更关键的是：如果我不满意，我可以直接改。

改 Markdown 原文 → 重新生成 → 刷新页面。这个循环快到让人上瘾。我不用再对着一份"差不多但不完美"的输出纠结，也不用重新训练模型让它理解我的意图。

更准确地说，我不是不用 AI 了，而是不再把最终成果锁死在一次性的图片、音频或 PPT 里。AI 仍然负责生成和设计，但最终产物回到了我熟悉、可编辑、可版本管理的 HTML 世界里。

我不再被模型左右了。我重新掌握了控制权。想调整哪个段落？改。想换一种呈现方式？换。输出什么结构、什么风格、什么深度——我自己说了算。

这种感觉，有点像做 web 开发时从"全栈框架"回到"裸写 HTML+CSS"——更自由、更可控、更爽。

为什么我选择 html-anything，而不是视频 / 播客 / PPT

这个选择背后是我对"知识表达"这件事的重新理解。

我们总以为知识需要"多媒体呈现"。信息图、播客、视频、PPT……好像形式越丰富，知识传播效果就越好。

但我现在的看法是：对绝大多数知识类内容来说，一个结构清晰、排版优雅的网页，比什么都强。

原因有三：

1. 网页是结构化的，而音频 / 视频是线性的。

一篇文章有标题、有小节、有代码块、有引用、有内部链接。读者可以跳读、回看、反复琢磨。而音频和视频必须按时间轴一帧一帧地过——你没法"跳读"一段播客。

2. 网页是可交互的。

读者可以点开链接查看引用来源，可以展开代码块复制代码，可以调整字号和配色。这些东西，一张"信息图"给不了。

3. 网页是活的。

我改 Markdown，html-anything 重新生成，网页就更新了。没有"重新渲染"的成本，没有"版本管理"的噩梦。我甚至可以保留 Git 历史，追踪每一次改动的脉络。

不是说视频和播客不好——我在做一些系统性总结时也会用 NotebookLM。但对于日常的知识沉淀和分享，网页是最轻量、最灵活、最可控的载体。

我的三层架构：把 Obsidian 变成"知识代码仓库"

说完了理念，聊聊落地。我目前的知识体系分三层：

第一层：Raw Sources（原始资料层）

位置：00_Inbox/网页剪藏/

我把剪藏的文章、PDF、会议记录、灵感碎片全丢在这里。原则是：只读不改，保留出处。

这层是证据层，不是阅读体验层。我不让任何 LLM 改写原始材料。可以摘要、引用、链接，但绝不覆盖原文。

这样做的好处是：任何时候我都能追溯到原始出处，不会被 LLM 的"润色"带偏。

第二层：Wiki（LLM + 人共同维护的知识层）

位置：02_学习/AI工具能力探索/、02_学习/AI Coding工作流/

这是知识库的核心。LLM 从原始材料中编译出结构化页面，我负责审核和调整。这些页面有：

  • 明确的 frontmatter（标签、状态、可信度）
  • 来源链接（指向 raw sources 的 wikilink）
  • 反向链接（关联到其他主题页）
  • 适用范围和边界（"这个方法论适用于什么场景"）
  • 待验证问题（哪些还没想清楚）

这层是"活的"。我每次有新判断，都会补充回来。LLM 每次有新材料进来，也会尝试更新相关页面。

第三层：HTML 输出（发布层）

位置：_html_exports/ → 05_成品归档/

当 Wiki 页面内容稳定了，我就用 html-anything 生成 HTML 成品。分两个阶段：

  • 临时输出到 _html_exports/，看效果、调结构
  • 满意了，归档到 05_成品归档/HTML/

关键原则：Markdown 是源文件，HTML 是编译产物。维护的入口永远是 Markdown。如果 HTML 有问题，回到 Markdown 改，重新生成，而不是手改 HTML。这样保证了 Git 历史清晰、修改可追溯。

真正跑起来，靠的是 index.md、log.md 和 lint

我一开始也以为，LLM Wiki 的关键是"让 AI 帮我总结文章"。

后来才意识到，这只是第一步。一个知识库能不能长期运转，不取决于第一次总结得多漂亮，而取决于三件很朴素的事：

1. index.md：先让 AI 知道有哪些东西

如果每次都让 AI 全库乱翻，它很快就会变慢、变贵、变不稳定。

所以我需要一个索引层。它不一定真的叫 index.md，但必须承担这个职责：

  • 这篇材料在哪里？
  • 它讲什么？
  • 属于哪个主题？
  • 适合更新哪一类 Wiki 页面？
  • 是否已经被处理过？

对我的 vault 来说，最小版 index 可以先长这样：

| 文件 | 主题 | 摘要 | 状态 |
|---|---|---|---|
| 00_Inbox/网页剪藏/不用 RAG，纯 Markdown 就能搞定.md | LLM Wiki | 介绍两层 Markdown 知识库实践 | 已编译 |
| 00_Inbox/网页剪藏/Obsidian+iCloud+Claude Code...md | Obsidian 工作流 | 介绍 Obsidian + iCloud + Claude Code 组合 | 已参考 |

它的价值不是给人看，而是给 AI 选材料。

2. log.md：让知识库记住自己怎么变成现在这样

只要开始让 AI 维护知识库，就一定会遇到一个问题：三周后你会忘记当初为什么这么改。

所以需要一个 log.md 或者更新记录。它记录的不是流水账，而是关键操作：

  • 哪些 raw source 被摄入？
  • 更新了哪些 Wiki 页面？
  • 哪些判断被新增、删除或改写？
  • 哪些问题仍然待验证？
  • 哪些内容进入了 HTML 成品输出？

这和写代码时写 commit message 很像。没有 log，知识库会变成一团"看起来都对，但不知道从哪来"的东西。

3. lint：定期让 AI 扫一遍知识库卫生

LLM Wiki 不是生成完就结束，它需要定期 lint。

我会重点检查几类问题：

  • 断链：[[某个页面]] 实际不存在。
  • 重复概念：同一个东西被写了两篇。
  • 孤立页面：没有任何页面链接到它。
  • 过期判断：工具版本、实践经验已经变化。
  • 未标注冲突：两个页面观点矛盾，但没有显式说明。
  • 未消化材料：Inbox 里长期堆着但没有进入 Wiki。

这一步很关键，因为 LLM Wiki 最大的风险不是"它不整理"，而是"它整理出一堆看起来很像知识、其实没人维护的页面"。

如果你想照着做，先跑这个最小闭环

我不建议一上来就搭完整系统。最小闭环只需要 6 步：

  1. 建一个 Inbox
    用来放原始材料。可以是 Obsidian 的 00_Inbox/网页剪藏/，也可以是普通文件夹。
  2. 选一个很窄的主题
    不要从"个人知识库"这种大词开始。先选一个主题，比如"LLM Wiki 怎么落地"。
  3. 收集 3-5 篇材料
    数量太少没法形成关联，数量太多第一次容易失控。
  4. 让 Claude Code / Codex 编译一篇 Wiki 页
    要求它不要复述原文，而是生成结构化页面：核心判断、证据、反例、边界、待验证问题。
  5. 人工校对并补来源
    这是人必须参与的地方。AI 负责重读和提炼，人负责判断和定稿。
  6. 用 html-anything 生成一个可阅读页面
    先输出到 _html_exports/，满意后再移动到 05_成品归档/HTML/。

跑完这一轮，你就会知道这套东西适不适合你。不要先买一堆工具，也不要先设计十层目录。先让它跑起来。

一个真实案例：我把 Claude Code 书稿编译成了网站

这套思路不是停留在脑子里。我已经拿自己的 Claude Code 教程做过一次完整实验。

我原来在 Obsidian 里写了一组文章，主题是《重新定义编码：Claude Code 实践与思考》。它本来只是一个 Markdown 书稿目录，分散在 vault 里：

  • 00｜导读：为什么现在必须认真学习 Claude Code
  • 01｜Claude Code 是什么，为什么它适合团队开发
  • 03｜部署、安装与基础配置：让 Claude Code 真正可用
  • 05｜为什么团队应该从 CLAUDE.md 开始
  • 06｜Superpower 与 SDD：把 Claude Code 变成团队规范开发引擎
  • 07｜Hooks：把规范开发变成自动化工作流
  • 08｜MCP：让 Claude Code 接入外部系统与真实生产工具
  • 09｜Multi-Agent：从单兵工具到团队级并行开发
  • 10｜综合实战：从零搭建团队规范 AI 编码工作流
  • 13｜附录：值得持续跟进的高质量资源

这套内容如果只待在 Obsidian 里，它对我有用，但对读者不友好。别人看不到目录结构，也不会知道应该从哪一篇开始读。

后来我把它生成了 HTML 网站，并发布到了 GitHub Pages：

  • 访问地址：《重新定义编码：Claude Code 实践与思考》

这个实验让我确认了三件事：

第一，Markdown 适合写作，但 HTML 更适合阅读。
Obsidian 里的 Markdown 是我的工作区，网页是读者的阅读区。两者不应该互相替代。

第二，系列内容一旦变成网页，结构价值会被放大。
目录、章节、跳转、内部链接，这些在 Obsidian 里是作者工具；到了网页里，就变成了读者导航。

第三，发布不是终点，而是新一轮迭代的开始。
只要源头仍然是 Markdown，我就可以继续改书稿、重新生成网站、保留版本历史。这和写代码非常像：先发布一个可用版本，再持续重构。

所以我现在更确信：LLM Wiki 不应该只停留在"我自己查资料更方便"。它最终应该能走到成品层，让知识真正被读到、被分享、被反馈。

这套组合拳给了我什么？

我认真想了一下，这套体系给我带来的最大变化不是"效率提升"，而是心态的变化。

以前看到好文章，我的第一反应是焦虑——"这篇太好了，但我没时间整理，要不先存着吧"。存着之后又愧疚——"又吃灰了，我真懒"。

现在看到好文章，我很放松。剪藏、写两句感受、打标签，五分钟完事。我知道我的 Inbox 有 LLM 帮我"消化"，我知道等我积累了足够的素材，Claude Code 会帮我提炼整合，我知道最终成果可以随时调整和美化。

我不再害怕知识过载了，因为我知道有一整套流水线在后面接着。

另一个变化是：我重新开始享受"写东西"这件事了。

以前写文章，总觉得要有"完整的输出"才能拿得出手——要写得透彻、结构完美、语言精炼。这个心理门槛太高了，导致我大部分时间都在"构思"而不是"动笔"。

现在我用 LLM 生成初稿、用 html-anything 生成可视化版本、用 Git 追踪每一次迭代——我可以在"还不太完美"的时候就发出去看效果，然后持续改进。就像写代码一样，先跑起来，再优化。

写在最后

Karpathy 提出 LLM Wiki 之后，网上涌现了很多实现方案——有 Obsidian 插件、有 Claude Code 插件、有独立的 CLI 工具。它们都很棒，但对我来说都有些"隔靴搔痒"。

我的知识库有自己的组织形式，我喜欢的模型和工具有自己的生态，我希望的输出方式也和别人不一样。每个知识工作者的"知识系统"都是高度个人化的——别人的方案可以参考，但不能照搬。

所以我选择了自己搭一套。

这篇文章不是什么"终极方案"，也不是什么"替代 RAG 的革命"。它只是在 LLM 时代，一个程序员用自己的方式，重新掌控自己的知识产出的故事。
